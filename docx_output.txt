Authentication — how it works on every endpoint
Every protected endpoint requires a Firebase JWT token sent in the request header:
// In every API request from the frontend:
headers: { 'Authorization': 'Bearer <firebase_id_token>' }
 
// In the backend middleware (middleware/auth.js):
// 1. Extract token from header
// 2. Verify with Firebase Admin SDK: admin.auth().verifyIdToken(token)
// 3. Attach decoded user to request: req.user = { uid, email }
// 4. All routes use req.user.uid to find the MongoDB user document
📁 Folder structure
Create these folders in your project: /routes (one file per group), /controllers (business logic), /middleware (auth), /services (Claude API calls, cron jobs)
GROUP 1 — Authentication  ( /api/auth )
These endpoints run once — during signup and login. Firebase handles the actual password/Google auth. These endpoints just sync the Firebase user into our MongoDB.
POST  /api/auth/register   — Create user in MongoDB after Firebase signup
Screen
Signup screen (first time only)
Triggered by
User completes Firebase signup — frontend calls this immediately after
Auth required
No — public endpoint
Business Logic — step by step
Check if a user with this firebaseUid already exists in MongoDB.
If yes — this is a re-login, not a new user. Return the existing user with isNewUser: false.
If no — create a new User document with the provided fields. Set onboardingComplete: false.
Return the new user object and isNewUser: true.
Frontend redirects to onboarding Step 1 when isNewUser is true.
Request Body (JSON)
firebaseUid  string — from Firebase  ✱ required
email  string  ✱ required
name  string — user's full name  ✱ required
Response (JSON)
user  Full user object from MongoDB
isNewUser  true — frontend uses this to redirect to onboarding
Error Responses
400  Missing required fields
409  User already exists with this email (different Firebase account)
⚠  Important Notes
Never store passwords — Firebase handles all authentication. We only store the firebaseUid as a reference.
This endpoint has NO auth middleware — it runs before the user has a token.
GET  /api/auth/me   — Get logged-in user's full profile
Screen
App startup — every time app loads
Triggered by
App launches and user is already logged in
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Extract firebaseUid from the verified JWT token (req.user.uid).
Find User in MongoDB by firebaseUid.
If user not found — return 404 (edge case: Firebase account exists but MongoDB record missing).
Return full user object.
Frontend checks onboardingComplete — if false, sends user to onboarding flow.
Request Body (JSON)
No request body needed
Response (JSON)
user  Full User document from MongoDB
onboardingComplete  boolean — if false, redirect to onboarding
Error Responses
401  No token or invalid token
404  Firebase user exists but no MongoDB record — prompt re-registration
GROUP 2 — Onboarding  ( /api/onboarding )
These endpoints save onboarding data step by step. The most important endpoint is /complete which triggers the first Claude call and generates Week 1 plan.
PATCH  /api/onboarding/profile   — Save onboarding steps 1–5 progressively
Screen
Onboarding Steps 1–5 (Basic Stats, Fitness Bg, Goals, Diet, Health)
Triggered by
User taps Next on each onboarding step
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Validate that all required fields for the given step number are present.
Use MongoDB updateOne with $set to patch only the fields for this step.
Do NOT overwrite fields from previous steps.
Return the updated user.
Frontend advances to the next step screen.
Request Body (JSON)
step  number 1–5 — which step is being saved  ✱ required
data  object — fields for that step  ✱ required
  Step 1: { heightCm, currentWeightKg, targetWeightKg, targetDate, dateOfBirth, gender }
  Step 2: { fitnessLevel, lastActivePeriod, equipmentAccess, daysPerWeek, preferredTime }
  Step 3: { primaryGoal, motivationEvent, urgencyLevel }
  Step 4: { dietType, foodRestrictions, supplementsTaken }
  Step 5: { injuries, conditions, exercisesToAvoid, doctorClearance }
Response (JSON)
user  Updated user object
nextStep  number — which step to show next
Error Responses
400  Missing required fields for the step
400  Invalid step number
⚠  Important Notes
Use PATCH not PUT. PATCH updates only what you send. PUT would overwrite the entire document.
Save each step immediately — if user closes the app mid-onboarding, their progress is not lost.
POST  /api/onboarding/complete   — Save baseline + generate Week 1 plan with Claude
Screen
Generating screen (after Step 5 confirm)
Triggered by
User taps "Build my plan" on the confirmation screen
Auth required
Yes — Firebase JWT token in header
🤖  CLAUDE API CALL — Call 01 or Call 02 — generates full Week 1 workout plan
Business Logic — step by step
Save baselineLifts, strongestMuscle, weakestMuscle to the User document.
Calculate dailyCalorieTarget and dailyProteinTarget using Harris-Benedict formula based on user profile.
Save calculated targets to User document.
Determine if user is "returning with baseline" (fitnessLevel=returning AND baselineLifts provided) OR "beginner/long-gap" (no baseline).
Build the Claude prompt: assemble full user profile + baseline data. Use Call 01 prompt for returning users, Call 02 for beginners.
Call Claude API. Parse the JSON response — it contains a 7-day workout plan array.
For each of the 7 days: create a Workout document in MongoDB with status "planned" and the exercises from Claude's response.
Set User.onboardingComplete = true and User.currentWeekNumber = 1.
Set User.isDiscoveryWeek = true if beginner (no baseline lifts provided).
Return the week plan and a welcome coach note to the frontend.
Request Body (JSON)
baselineLifts  object — { chestPressKg, shoulderPressKg, squatKg } — may be null for beginners
strongestMuscle  string — e.g. "chest"
weakestMuscle  string — e.g. "legs"
Response (JSON)
weekPlan  Array of 7 workout documents created in DB
user  Updated user with onboardingComplete: true
coachNote  String — Claude's welcome message to the user
Error Responses
500  Claude API call failed — return a default template plan as fallback
400  User profile incomplete — some required onboarding field missing
⚠  Important Notes
This is the most critical endpoint in the app. If Claude fails, have a hardcoded fallback plan ready so the user is never stuck on the generating screen.
Harris-Benedict formula for calorie target: Men: 88.36 + (13.4 × weightKg) + (4.8 × heightCm) − (5.7 × age). Multiply by 1.55 for moderately active. Subtract 500 for fat loss goal.
Claude's response MUST be valid JSON. If parsing fails, retry once, then use fallback.
The 7 Workout documents created here are the backbone of the entire first week.
GROUP 3 — Workouts  ( /api/workouts )
The most complex group. Handles loading today's plan, logging sets in real time, completing a session, and modifying tomorrow's plan.
GET  /api/workouts/today   — Get today's workout plan
Screen
Workout Tab — when user opens the tab
Triggered by
User opens the Workout tab or app loads
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Get today's date (midnight to midnight in user's timezone).
Find the Workout document for this userId + today's date.
If no workout found for today — check if today is a rest day in the user's weekly split.
If rest day: return isRestDay: true.
If workout exists: populate exercise details by joining with Exercise collection (to get gifUrl, whyLabel).
Return the workout document with populated exercise data.
Request Body (JSON)
No request body needed
Response (JSON)
workout  Full Workout document for today including all exercises and sets
isRestDay  boolean — if true, show rest day screen
previousSession  Last completed session's summary (for context display)
Error Responses
404  No workout planned for today — frontend shows a "Contact your coach" prompt
PATCH  /api/workouts/:workoutId/log-set   — Log a single set in real time
Screen
Workout Tab — as user enters reps/weight for each set
Triggered by
User types actual reps or weight into a set row and moves to next field
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Find the Workout document by workoutId.
Use MongoDB positional operator to update the specific set: workout.exercises[exerciseIndex].sets[setIndex].
Update only the fields provided (actualReps, actualWeight, completed) — do not touch planned values.
If workout status is still "planned", update it to "in_progress".
Return success.
Request Body (JSON)
exerciseIndex  number — which exercise in the array (0-based)  ✱ required
setIndex  number — which set (0-based)  ✱ required
actualReps  number — what user actually did
actualWeight  number — in kg
completed  boolean — mark this set as done
Response (JSON)
success  true
updatedSet  The updated set object
Error Responses
404  Workout not found
400  Invalid exerciseIndex or setIndex
⚠  Important Notes
This endpoint fires very frequently — every time user enters a number. Keep it fast. No Claude call here.
Never overwrite plannedReps or plannedWeight. These are the baseline for pattern analysis.

POST  /api/workouts/:workoutId/complete   — Mark session complete — triggers AI summary + tomorrow's plan
Screen
Workout Tab — after user taps Mark Complete
Triggered by
User taps "Mark Session Complete" button
Auth required
Yes — Firebase JWT token in header
🤖  CLAUDE API CALL — Call 03 — post-session summary + tomorrow's plan generation
Business Logic — step by step
Update the Workout document: status = completed, sessionRating, sessionFeel, sessionDurationMins.
Detect Personal Records: for each completed exercise, check if actualWeight × actualReps beats the user's previous best (query last 30 days of workouts). Save new PRs.
Build the context payload for Claude: user profile + last 3 days of workouts (planned vs actual) + today's session + session rating.
Call Claude API (Call 03). Parse response: { summary, tomorrow: { exercises[] } }.
Create a new Workout document for tomorrow with the exercises from Claude's response and status: planned.
Save Claude's summary text to today's workout.aiSummary field.
Update Metric document for this week: increment sessionsCompleted, save any new PRs.
Return everything to the frontend.
Request Body (JSON)
sessionRating  number 1–10  ✱ required
sessionFeel  string: Easy / Good / Hard / Exhausted  ✱ required
sessionDurationMins  number — optional
Response (JSON)
workout  Updated workout with status: completed
aiSummary  Claude's honest post-session feedback string
tomorrowPlan  Full workout document for tomorrow (preview)
newPRs  Array of any personal records set today
Error Responses
500  Claude API failed — return success for session completion but skip AI summary. Show "Summary unavailable" in UI.
⚠  Important Notes
Split this into two steps: first save the completion data to DB immediately, THEN call Claude. User should never wait for Claude to see their session saved.
The tomorrow Workout document created here has status "planned". It becomes the plan the user sees next time they open the Workout tab.
If a Workout document for tomorrow already exists (user had a pre-existing plan), OVERWRITE it with Claude's new suggestion — but only after user confirms on the preview screen.

recoveryWarning  string or null — warning if muscle was trained recently

Do not overwrite tomorrow's plan until user confirms. Return preview only.

POST  /api/diet/analyze-photo   — Upload food photo and get AI nutrition breakdown
Screen
Diet Tab — food photo upload
Triggered by
User taps camera icon and uploads a photo
Auth required
Yes — Firebase JWT token in header
🤖  CLAUDE API CALL — Call 06 — Claude Vision food photo analysis
Business Logic — step by step
Receive the base64 image. Upload it to Cloudflare R2: save under /food-logs/{userId}/{date}/{mealType}.jpg.
Get the user's diet type (eggetarian etc.) and today's running totals from DietLog.
Build Claude Vision prompt (Call 06): image + meal type + diet type + calories consumed so far + protein consumed so far.
Call Claude API with the image. Parse the JSON array of food items.
Return items with confidence levels, totals, and the one-line tip.
Do NOT save to DietLog yet — user must confirm the items first (next endpoint).
Request Body (JSON)
photo  base64 encoded image string  ✱ required
mealType  string: breakfast / lunch / snack / dinner  ✱ required
Response (JSON)
items  Array of detected food items with name, quantityG, calories, proteinG, carbsG, fatG, confidence
totalCalories  number — sum of all items
totalProteinG  number — sum of all items
oneTip  string — Claude's one-line tip based on remaining targets today
photoUrl  string — Cloudflare R2 URL where photo was saved
⚠  Important Notes
Upload photo to R2 BEFORE calling Claude — you need the URL to store in the meal later.
This is the most expensive Claude call per token. Keep the context lean — only diet profile + today's totals. Do NOT send full workout history here.
If Claude returns low confidence on an item, the frontend shows a "?" icon — user can edit.
POST  /api/diet/log-meal   — Confirm and save a meal to today's diet log
Screen
Diet Tab — after user confirms food items
Triggered by
User reviews AI-detected items, optionally edits, then taps Confirm
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Find or create today's DietLog for this user.
Calculate totals for this meal by summing all item values.
Mark any item where user changed name/quantity as editedByUser: true.
Push the new meal object into dietLog.meals array.
Recalculate and update dietLog.totalCalories, totalProteinG, totalCarbsG, totalFatG.
Check if this is the 2nd meal logged today — if yes, trigger the daily coach tip (Call 07) asynchronously.
Return updated DietLog and running totals.
Request Body (JSON)
mealType  string: breakfast / lunch / snack / dinner  ✱ required
photoUrl  string — R2 URL from the analyze-photo response  ✱ required
items  Array of confirmed food items (may be edited by user)  ✱ required
aiTip  string — the tip from the analyze-photo response to store
Response (JSON)
dietLog  Updated DietLog with new meal added
runningTotals  Updated totals for the day
⚠  Important Notes
Call 07 (daily nutrition tip) must be triggered asynchronously — do not make the user wait for it. Use a background process or fire-and-forget.
If the same meal type (e.g. breakfast) already exists in today's log, append as a new entry — do not overwrite. A user can eat breakfast twice.

GET  /api/diet/tip   — Get or generate today's daily nutrition coach tip
Screen
Diet Tab — daily coach tip card
Triggered by
Automatically after 2nd meal is logged, or when user opens Diet tab
Auth required
Yes — Firebase JWT token in header

Business Logic — step by step
Check today's DietLog for dailyCoachTip and dailyTipGeneratedAt.
If tip was already generated today (dailyTipGeneratedAt is today): return the cached tip. Do NOT call Claude again.

GROUP 5 — Coach Chat  ( /api/coach )
The most powerful endpoint. Full conversational Claude access with complete user context injected. Handles all user questions and the "modify tomorrow" flow.
POST  /api/coach/chat   — Send a message and get AI coach response
Screen
Coach Chat Tab
Triggered by
User sends any message in the chat interface
Auth required
Yes — Firebase JWT token in header
🤖  CLAUDE API CALL — Call 08 — full coach chat with streaming response
Business Logic — step by step
Build the full base context payload: user profile + aggregated stats + last 3 days workouts + last 3 days diet + today's context + pattern summaries (Week 2+).
If intent is "modify_tomorrow": also fetch tomorrow's current Workout document and inject it into the prompt. Instruct Claude to output a workout-plan JSON block.
Append the user's message to the conversation history.
Call Claude API (Call 08) with the full system prompt + conversation history.
Receive streaming response. Detect if response contains a ```workout-plan JSON block.
If workout plan detected: parse the JSON, store it temporarily (in-memory or Redis with 30min TTL) with a pendingPlanId.
Return the full reply text and pendingPlanId if a plan was detected.
Frontend renders the reply. If pendingPlanId exists, renders the plan preview card with Confirm and Change More buttons.
Request Body (JSON)
message  string — user's message  ✱ required
conversationHistory  array of last 6 messages: [{ role, content }]  ✱ required
intent  string (optional): "modify_tomorrow" — detected by frontend keyword matching
Response (JSON)
reply  string — Claude's full response
workoutPlanJson  object or null — if Claude returned a workout plan block
pendingPlanId  string or null — temp ID for the unsaved plan preview
Error Responses
500  Claude API failed — return a fallback message: "Your coach is unavailable right now. Try again in a moment."
⚠  Important Notes
The conversationHistory sent from frontend should be the last 6 messages MAX. Never send the full chat history — it will exceed token limits.
The pending plan is NOT saved to MongoDB until the user confirms (next endpoint). Store it temporarily.
Streaming: ideally use Server-Sent Events (SSE) to stream Claude's response token by token. This makes the chat feel instant. If SSE is too complex for now, use regular POST and wait for full response.
POST  /api/coach/confirm-plan   — Save a coach-chat-modified plan to DB
Screen
Coach Chat Tab — after user taps "Save this plan" on preview card
Triggered by
User taps Confirm on the workout preview card in chat
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Retrieve the pending plan from temporary storage using pendingPlanId.
If not found (expired): return 404 — frontend prompts user to regenerate.
Find the Workout document for targetDate.
Replace exercises array with the confirmed plan. Set planSource: "coach_chat".
Delete the pending plan from temporary storage.
Return the saved workout.
Request Body (JSON)
pendingPlanId  string — temp ID from the chat response  ✱ required
targetDate  ISO date string — which day this plan is for  ✱ required
Response (JSON)
workout  Saved Workout document
message  string: "Tomorrow's plan saved successfully"
⚠  Important Notes
pendingPlanId expires after 30 minutes. If user takes too long the plan is gone — they need to ask Claude again.
GROUP 6 — Exercise Library  ( /api/exercises )
These endpoints power the Add Exercise browser and the exercise detail screen. Read-only — this is master data, not user data.
GET  /api/exercises   — Browse exercises with filters
Screen
Add Exercise screen — browse by muscle group
Triggered by
User opens Add Exercise screen or taps a muscle group
Auth required
Yes — Firebase JWT token in header
Business Logic — step by step
Accept query parameters: muscleGroup, equipment, goalTag, search (text), page, limit.
If search is provided: do a case-insensitive regex match on exercise name and goalTags array.
If muscleGroup is provided: filter by that group.
Filter out exercises where avoidIf array contains any of the user's injuries.
Sort by name alphabetically.
Paginate: default 20 per page.
Return matching exercises with gifUrl, whyLabel, equipment, muscleGroup.
Request Body (JSON)
No request body needed
Response (JSON)
exercises  Array of exercise objects matching the filters
total  number — total matching count
⚠  Important Notes
Always filter by user's injuries — a user with plantar_fasciitis should never see jumping exercises.
The goalTag filter enables the "reduce belly fat" search without vector DB. e.g. goalTag=belly_fat returns all exercises tagged with that goal.

POST  /api/progress/weight-checkin   — Log Monday weight and generate weekly report
Screen
Progress Tab — Monday weight check-in prompt
Triggered by
User enters their weight on the Monday check-in card
Auth required
Yes — Firebase JWT token in header
🤖  CLAUDE API CALL — Call 09 — weekly pattern insight generation (if Week 2+)
Business Logic — step by step
Find or create this week's Metric document.
Save weightKg.
Find last week's Metric to calculate weightDeltaKg.
Update User.currentWeightKg.
If weekNumber >= 2: fetch last 4 WeeklySummary documents and call Claude (Call 09) to generate pattern insights.
Save patternInsights to this week's WeeklySummary.
Check and award any new milestone badges: first_kg_lost, back_to_start_weight, 5_sessions_week, etc.

GROUP 8 — Background Cron Jobs  ( server-side only )
These are NOT HTTP endpoints that the frontend calls. They run automatically on a schedule inside the backend server using the node-cron library. Frontend never touches these.
📦 Setup
npm install node-cron   then import in your main server.js file. Cron jobs start when the server starts.

Business Logic — step by step
Get all users where onboardingComplete: true.
For each user (run sequentially with a small delay to avoid Claude rate limits):
Fetch all Workout documents for this user for the past 7 days.
Fetch all DietLog documents for the past 7 days.
Build the compression prompt (Call 10): raw week data as JSON.
Call Claude API. Parse the structured summary JSON.
Create or upsert a WeeklySummary document with the compressed data.
Store: sessionsCompleted, peakLifts, plannedVsActual deltas, feelPattern, avgProtein, dietPattern.
Log completion and any errors.
⚠  Important Notes
Add a 2-second delay between each user's Claude call to avoid hitting rate limits.
If Claude fails for a user: log the error and continue to next user. Do not let one failure stop the whole job.
node-cron syntax for Sunday 11:30 PM:  cron.schedule("30 23 * * 0", () => { runWeeklyCompression(); })
POST  CRON — Monday 6:00 AM   — Generate Week N+1 plan for all users
Screen
No screen — users see the new plan when they open the app Monday morning
Triggered by
Every Monday at 6:00 AM automatically
Auth required
No — public endpoint
🤖  CLAUDE API CALL — Call 11 — generate next week's full plan
Business Logic — step by step
Get all users where onboardingComplete: true.
For each user:
Fetch the last 4 WeeklySummary documents (just created by Sunday's cron).
Fetch user's latest weight from Metric (Monday check-in if done, else last known).
Build the planning prompt (Call 11): summaries + profile + weight trend.
Call Claude API. Parse the 7-day plan JSON.
Create 7 new Workout documents (Mon–Sun) with status "planned".
Increment User.currentWeekNumber.
Send a push notification: "Your Week N plan is ready 💪"
⚠  Important Notes
node-cron syntax for Monday 6:00 AM:  cron.schedule("0 6 * * 1", () => { runWeeklyPlanGeneration(); })
If a user has not done their Monday weight check-in yet, use their last known weight. Do not skip them.
The push notification requires Firebase Cloud Messaging (FCM) — set this up separately.
